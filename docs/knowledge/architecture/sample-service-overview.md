# Sample — Service overview

> **This is a sample document** included with the template to show the expected shape of an architecture note. Replace it with a real overview, or delete it once your bundle has real content.

## Context

One or two paragraphs that set the stage. What is the system, what problem does it solve, and what constraints shape its structure? Link out to `README.md` or external docs for the user-facing description — this document is for *internal* structural understanding.

## Components

A short list of the services / modules / processes that make up the system, with one-line summaries. Each entry should link to a more detailed document when one exists.

* **Component A** — what it does, and the one thing a reader most needs to know about it.
* **Component B** — same.

## Data flow

How requests, events, or jobs move through the components. A small diagram (Mermaid, ASCII, or a linked image) usually beats prose here.

## Operational implications

What this structure makes easy, and what it makes hard. Anything that would surprise a new contributor goes here.

## Related

* Bundle-internal cross-links go here, e.g. `[ADR-XXXX](/docs/knowledge/adr/XXXX-...md)`.
* External references (RFCs, vendor docs, post-mortems) go here.
